1. Небольшой API ядра, специально для сервисов.
2. микросообщения для общения ядра и сервисов ( ничего общего с
   сообщениями процесс-сервис).

Всё это для того, чтобы максимально отвязать сервисы от ядра и по
возможности от его адресного пространства. Также для них можно понизить
приоритет сегментов. Ядро будет несколько поболе, но система будет стабильней.
Ещё можно породить несколько служебных процессов в самом ядре.

При такой структуре планирование, всё же должно быть в ядре, а не в отдельном
сервисе. Хотя конечно же сервисам можно позволить влиять на планирование.
Например при помощи введения в ядро АПИ типа _начало_непрерываемого_кода(),
_конец_непрерываемого_кода(). Но надо быть предельно осторожным, т.к. сервис
может не вызвать _конец_непрерываемого_кода().
